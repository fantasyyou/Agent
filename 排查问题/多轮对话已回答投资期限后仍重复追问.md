# 多轮对话已回答投资期限后仍重复追问

对应截图：[多轮对话已回答投资期限后仍重复追问.png](./多轮对话已回答投资期限后仍重复追问.png)

## 问题现象

多轮理财需求收集过程中出现两种同类问题：

```text
客服：这笔资金预计多长时间内不会使用？
用户：未来3个月基本不会动用。
客服：您计划投入的大致金额是多少？
用户：20万。
客服：这笔资金预计多长时间内不会使用？
```

以及：

```text
客服：您计划投入的大致金额是多少？
用户：30万。
客服：您计划投入的大致金额是多少？
```

系统没有稳定地将用户对当前问题的回答保存到对应槽位，所以继续把期限或金额判断为缺失并重复询问。

## 排查过程

### 1. 查询 Redis 对话状态

```bash
docker exec \
  -e REDISCLI_AUTH=redis \
  agent-redis \
  redis-cli --scan --pattern 'agent:dialogue:*'
```

使用查到的 Key 获取状态：

```bash
docker exec \
  -e REDISCLI_AUTH=redis \
  agent-redis \
  redis-cli GET 'agent:dialogue:用户ID:会话ID'
```

实际状态为：

```json
{
  "intent": "product_recommendation",
  "slots": {
    "liquidity_requirement": "low",
    "risk_tolerance": "conservative"
  },
  "status": "need_clarification",
  "last_question": "这笔资金预计多长时间内不会使用？"
}
```

状态中缺少：

```json
{
  "investment_period_months": 3,
  "investment_amount": 200000
}
```

由此确认 Redis 没有主动丢失数据，而是 Python 返回给 Redis 的有效槽位中原本就没有期限和金额。

### 2. 检查模型输入

原提取器只把用户本轮文本传给模型。例如用户回答“30万”时，模型没有看到上一问是投资金额，也没有看到已经确认的风险偏好和期限。

孤立的“30万”可能表示投资金额、资产、收入或贷款金额。模型可能返回空 `slots`，也可能返回无法通过校验的字段和值。

### 3. 检查字段校验

原代码对整数字段要求过严：

```python
elif definition.value_type == SLOT_VALUE_TYPE_INTEGER:
    valid_type = (
        isinstance(value, int)
        and not isinstance(value, bool)
        and value > 0
    )
```

protobuf传输后的整数可能变为 `3.0`，模型也可能返回 `"3个月"` 或 `"20万"`。这些值都会被原代码过滤。

### 4. 检查下一问策略

原策略为了避免重复问题，会降低上一问题的优先级：

```python
if last_question and last_question == slot.question:
    value -= 1000
```

当期限没有提取成功时，系统没有继续澄清期限，而是跳去询问金额；金额也失败后又跳回期限，于是形成来回追问。

## 根因

问题由三点共同导致：

1. 模型只看到用户本轮的孤立回答，缺少上一问和已有槽位；
2. 数字格式校验过严，`3.0`、`3个月`和`20万`等值会被过滤；
3. 字段提取失败后，提问策略切换到其他字段，掩盖了真实失败。

## 解决方案

### 1. 给模型传递必要上下文

修复前：

```python
extraction = self.extractor.extract(text, candidate_workflows)
```

修复后：

```python
extraction = self.extractor.extract(
    text,
    candidate_workflows,
    existing_slots=existing_slots or {},
    last_question=last_question,
)
```

模型现在能够看到：

```text
已经确认的参数
客服上一问
用户本轮回答
```

因此能将“30万”理解为对投资金额问题的回答。

### 2. 统一金额和期限格式

修复前只校验原始类型，修复后先标准化再校验：

```python
slots = {}
for key, value in combined_slots.items():
    definition = definitions.get(key)
    if definition is None:
        continue
    valid, normalized = _normalize_slot_value(definition, value)
    if valid:
        slots[key] = normalized
```

支持的常见转换：

```text
3.0       → 3
3个月     → 3
半年      → 6
1年       → 12
20万      → 200000
30万元    → 300000
200000元  → 200000
```

### 3. 模型漏提取时使用确定性兜底

即使模型返回空 `slots`，如果系统上一轮明确询问金额或期限，也会直接从用户本轮原话中解析对应字段：

```python
expected_definition = next(
    (
        definition
        for definition in workflow.slots
        if last_question and definition.question == last_question
    ),
    None,
)
if expected_definition is not None:
    answered_value = _extract_answer_value(expected_definition, text)
    if answered_value is not None:
        combined_slots[expected_definition.name] = answered_value
```

例如：

```text
上一问是投资金额 + 用户回答30万
→ investment_amount=300000

上一问是投资期限 + 用户回答未来3个月不会使用
→ investment_period_months=3
```

### 4. 无效回答继续澄清当前字段

修复后，上一问题对应的字段仍然缺失时，不再切换到其他问题：

```python
for candidate in candidates:
    if last_question and last_question == candidate.question:
        return candidate
```

只有当前字段成功取得有效值并从缺失列表移除后，系统才会询问下一个字段。

## 验证

Python测试：

```bash
cd agent-python
python -m unittest discover -s tests -v
```

当前17个单元测试通过，包含：

- `3个月`转换为`3`；
- protobuf返回的`3.0`转换为`3`；
- `20万`转换为`200000`；
- 模型返回空槽位时仍能解析`30万元`；
- 模型返回空槽位时仍能解析`未来3个月`；
- 无效回答不会在期限和金额之间来回跳转。

重新部署后，清除旧的测试状态，再执行：

```text
第一轮：最近账户里有一笔暂时不用的资金，我不太能接受净值出现明显起伏，想做一个偏保守的配置。
第二轮：从现在算，未来3个月基本不会动用。
第三轮：30万。
```

通过标准：

1. 第二轮后 Redis 中存在 `investment_period_months=3`；
2. 第三轮得到 `investment_amount=300000`；
3. 系统不再重复询问期限或金额；
4. 必填字段完整后结束流程并删除 Redis 短期状态。

## 状态模型改造

当前任务工作区已增加 `active_slot、slot_status、evidence、retry_count`。Go负责合并和持久化最终状态，Python只返回本轮 `slot_updates、status、active_slot、next_action`；`last_question`只作为模型理解上下文和展示文本，不再是识别当前字段的唯一依据。
